# Technical Audit Report: HackItAll - Opportunity Access Platform

## 1. Overview of the Repository Architecture

The HackItAll repository implements an AI-powered opportunity discovery platform. The architecture is primarily composed of:

*   **Backend:** A FastAPI-based RESTful API written in Python. It is organized into distinct directories for routing (`api/`), business logic (`services/`), database models (`models/`), and middleware (`middleware/`). It uses SQLAlchemy as an ORM with an SQLite database (configured to be easily switchable to PostgreSQL). Authentication is handled externally by Clerk.
*   **Frontend Web:** A React 18 application built with Vite (`web/`), using React Router for navigation and Framer Motion/GSAP for animations.
*   **Frontend Mobile:** A React Native application built with Expo (`mobile/`), using React Navigation and AsyncStorage.
*   **AI/ML Layer:** A recommendation engine (`services/recommendation_service.py`) and a local AI extractor for scraping opportunities (`services/nlp/local_extractor.py`, `services/scraper/dispatcher.py`) leveraging LLMs (specifically referenced as `gemma3:4b` using Ollama).
*   **Notification System:** A backend service (`services/notification_service.py`) scheduled via a background task (`scheduler.py`) to handle reminders.

## 3. Missing Features

Based on the specification:

*   **Active SMS/Email Integration:** The notification service is a skeleton. It lacks integration with an actual email provider (e.g., SendGrid, AWS SES) or SMS provider (e.g., Twilio).
*   **Frontend Low-Bandwidth Support:** While the backend exposes a low-bandwidth API toggle, the frontend applications (`web/` and `mobile/`) do not seem to have a dedicated "lightweight UI" mode or logic to consume the `LowBandwidthService` headers effectively.
*   **Persistent AI Summaries:** The specification asks for "Why this fits you" and "How to apply" text generated by GenAI for *each* opportunity. Currently, the AI logic is heavily skewed towards scraping extraction (`UniversalLocalScraper`) and dynamic skill-gap generation, rather than pre-computing and storing these specific explanations for all opportunities.

## 4. Implementation Deviations

*   **Scraping Architecture:** The implementation includes a robust scraping engine (`services/scraper/`) utilizing BeautifulSoup, Playwright, and local LLMs (Ollama/Gemma) to ingest opportunities dynamically. This is an advanced addition that goes beyond simple "aggregation."
*   **Authentication:** The system delegates authentication to Clerk (`api/auth.py`), utilizing Clerk JWTs for session management instead of managing passwords and sessions locally, which is a modern architectural choice but deviates from a purely self-contained system.
*   **Recommendation Caching:** The `RecommendationEngine` uses a local file-based JSON cache (`.recommendation_cache.json`) instead of the existing Redis cache (`services/cache_service.py`).
*   **Service Layer Bleed:** Some API endpoints, such as `/tracked/scrape` in `api/tracking.py` and `/opportunities/scrape-url` in `api/opportunity.py`, contain direct database operations and orchestration logic that should ideally reside in the service layer.

## 5. Suggested Improvements needed to match original design

1.  **Integrate Real Notification Providers:** Replace the `print` stubs in `services/notification_service.py` with actual API calls to SendGrid (or similar) for emails and Twilio for SMS.
2.  **Implement Frontend Low-Bandwidth UI:** Update the React and React Native applications to detect the user's `low_bandwidth_mode` preference and render a simplified, text-heavy interface that avoids loading images or complex animations (like those provided by `framer-motion` and `three` in the web app).
3.  **Refine AI Explanations:** Update the scraping or ingestion pipeline to explicitly generate and store the "Why this fits you" and "How to apply" fields on the `Opportunity` model, ensuring they are readily available for the frontend without real-time AI inference.
4.  **Standardize Service Layer:** Refactor API endpoints to remove direct SQLAlchemy queries (`db.query(...)` and `db.add(...)`). Move this logic entirely into the respective service classes (e.g., `TrackerService`, `ScraperService`) to maintain a clean MVC/Service-oriented architecture.
5.  **Unify Caching:** Migrate the local JSON file caching in the `RecommendationEngine` to use the centralized Redis `CacheService` to ensure scalability and consistency across the backend.
